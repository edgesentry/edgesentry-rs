# CV 統合：システム境界と制約

edgesentry-rs が受け取るのは**エンティティの位置情報**です。映像フレームは処理しません。

このドキュメントでは、edgesentry-rs が本番パイプライン全体のどこに位置するか、
ingest 層が入力に対して何を前提とするか、
そして上流 CV 層から伝播する環境的制約について定義します。

---

## パイプライン上の位置

```
カメラフレーム  (edgesentry-rs の範囲外)
      │
      ▼
CV モデル — 物体検出 + 追跡  (edgesentry-rs の範囲外)
      │
      │  EntityFrame: { id, class, x_m, y_m, vx_ms, vy_ms, timestamp_ms }
      │  オプション:   { confidence: f32 }
      ▼
edgesentry-ingest  ◄── edgesentry-rs の境界はここから始まる
      │
edgesentry-compute → edgesentry-evaluate → edgesentry-assess
      │
edgesentry-explain → edgesentry-audit → R2 Object Lock
```

ingest adapter の責務：

1. エンティティ位置の受信（UDP ソケット、ファイル、AIS ストリーム、CV adapter）
2. サイト固有のピクセル座標または地理座標をメートルに変換
3. エンティティクラス名を `EntityClass` 型に正規化
4. `EntityFrame` JSONL を出力

ingest より下流はすべてメートル座標で動作し、サイト非依存です。

---

## edgesentry-rs がエンティティ位置に対して前提とすること

| フィールド | 前提 |
|---|---|
| `x`, `y` | サイト定義の基準点を原点とした実世界メートル値 |
| `vx`, `vy` | メートル毎秒。フレーム間差分または CV トラッカーの出力 |
| `class` | `Forklift`, `Person`, `Vessel`, `ReachStacker`, `AisGap` のいずれか |
| `timestamp_ms` | Unix ミリ秒、エンティティごとに単調増加 |
| 精度 | 運用距離での位置誤差 < 0.5 m（TTC が意味を持つために必要） |

CV 層がノイズの多い位置（ジッタ、フレーム間のテレポート）を出力した場合、
物理計算が誤った TTC および制動距離を算出します。
edgesentry-rs にはこの上流ノイズを検出・補正する手段がありません。

## サイト適応：プロファイルシステム

サイトごとにカメラ構成が異なります。
`profile/` ディレクトリがサイトごとの適応層です。

```toml
# profiles/my-site/params.toml
[reference_point]
lat_deg = 1.2640      # AIS / GPS 入力用
lon_deg = 103.8200

[camera.zone_b]
px_per_metre = 42.3
origin_px = [960, 540]

[ais_gap]
threshold_s = 480
```

物理ルールはピクセルではなくメートルを参照します。
キャリブレーション（ピクセル → メートル変換）は params.toml の値を使用して
ingest adapter が行います。

rules.json のゾーンポリゴンはサイト基準点からのメートル座標で定義します：

```json
{
  "rule_id": "RESTRICTED_ZONE",
  "condition": "zone_member",
  "zone": [[-300, 200], [300, 200], [300, 700], [-300, 700]]
}
```

## 環境的制約とその伝播

edgesentry-rs は正しい入力が与えられれば決定論的に動作します。
CV 層からの誤った入力は以下のように伝播します。

### 1. エンティティが検出されない（低照度・遮蔽・霧）

エンティティが EntityFrame に現れません。
`RiskEvent` は生成されません。
監査チェーンには沈黙が記録されます——「何も起きなかった」と区別できません。

**対策：** 検出信頼度がしきい値を下回った場合、CV adapter から `CoverageGap`
イベントを出力します。このイベントが RiskEvent と並んでチェーンに封印され、
監視が劣化していた時間帯を監査可能にします。

### 2. 信頼度の低い検出（影・グレア）

CV トラッカーが低信頼度で位置を出力する場合があります。
現在の `Entity` 構造体には confidence フィールドがないため、
edgesentry-rs はそのまま受け入れます。

**予定：** `Entity` に `confidence: Option<f32>` を追加（#321）。
evaluate 層でしきい値以下のイベントを抑制し、
監査レコードに信頼度スコアを含めることができるようになります。

### 3. 座標ノイズ・ジッタ

ノイズの多い位置は速度ベクトルの不安定化を引き起こします。
誤った TTC アラートの発生や実際の危険の見逃しにつながります。

**対策：** EntityFrame 出力前に ingest adapter 内でカルマンフィルタまたは
指数移動平均を適用します。edgesentry-rs はフィルタリングを行いません——
フィルタリングは ingest 層の責務です。

### 4. タイムスタンプのスキュー（熱スロットリング・フレームドロップ）

エッジデバイスが熱負荷によりフレームをドロップすると、タイムスタンプが不規則になります。
TTC 計算はフレーム単位で正確ですが、
2つのエンティティが実際に最接近した瞬間を見逃す可能性があります。

**対策：** 処理スレッドのソフトウェアタイムスタンプではなく、
ハードウェアタイムスタンプ（カメラの PPS または NTP 同期）を使用します。

## 監査チェーンが保証すること（しないこと）

### 保証すること

- 生成されたすべての `RiskEvent` は、封印後に改ざん・削除・並び替えされていない
  （BLAKE3 ハッシュ連鎖 + Ed25519 署名）。
- チェーンはオペレーターを介さずに第三者が検証できる。

### 保証しないこと

- すべての危険なイベントが CV 層で*検出*されたこと。
- カメラが適切な方向を向いていたこと。
- CV モデルが精度の保証範囲内で動作していたこと。

監査チェーンは、edgesentry-rs が「伝えられたこと」の改ざん防止記録です。
完全な状況把握の証明ではありません。

## Phase 1 の範囲（POC）

Phase 1 では CSV フィクスチャのリプレイ（`eds ingest replay`）を使用します。
フィクスチャはキャリブレーション済みの CV adapter が出力するものと等価な、
事前計算済みのエンティティ位置を表します。

Phase 1 にライブカメラ統合はありません。
デモでは物理エンジン、ルール評価、監査チェーンを検証します。

CV 統合は Phase 2 の成果物です。
